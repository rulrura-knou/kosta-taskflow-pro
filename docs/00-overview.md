## 읽는 순서

```
00-overview → 01-product → 02-specs → 03-design → 04-tasks → 05-conventions
```

| 단계 | 이유 |
|---|---|
| `00` 먼저 | 전체 지도를 파악하고 나머지 문서의 맥락을 잡는다 |
| `01` 다음 | "무엇을 만드는가"를 알아야 기술 선택의 이유를 이해할 수 있다 |
| `02` 다음 | 기능 요구사항을 알고 난 뒤 기술 명세를 읽어야 설계 의도가 보인다 |

## 왜 이렇게 나누는가 — 관심사 분리 원칙

소프트웨어에서 관심사 분리(Separation of Concerns)란 **서로 다른 이유로 변경되는 것은 서로 다른 곳에 둔다**는 원칙이다.

이 문서 구조도 같은 원칙을 따른다.

| 관심사 | 변경 이유 | 담당 문서 |
|---|---|---|
| 제품 방향 | 비즈니스 요구가 바뀔 때 | `01-product.md` |
| 기술 구현 | 스택이나 API 설계가 바뀔 때 | `02-specs.md` |
| 사용자 경험 | 화면 흐름이나 디자인이 바뀔 때 | `03-design.md` |
| 진행 상황 | 작업이 추가·완료·취소될 때 | `04-tasks.md` |
| 협업 방식 | 팀 규칙이 바뀔 때 | `05-conventions.md` |

한 문서에 모두 몰아넣으면 **제품 기획이 바뀔 때 기술 명세 문서까지 뒤지게 된다**.
분리하면 변경이 필요한 문서만 열면 되고, 나머지는 건드리지 않아도 된다.

> 코드에서 단일 책임 원칙(SRP)이 클래스를 분리하듯,
> 문서에서는 관심사 분리 원칙이 파일을 나눈다.
